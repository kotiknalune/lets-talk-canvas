What is CANVAS

What is canvas...

canvas element was introduced in late 2014

Canvas is a 2D drawing API added to documentation in late 2014 and in our editor it will look something like this...

HTML / CSS


// CSS
body { margin: 0; }
canvas { width: 100%; height: 100%; }
 
// HTML

JS


// JS
const canvas = document.getElementById('canvas');
const ctx = canvas.getContext("2d");
ctx.beginPath();
ctx.arc(95, 50, 40, 0, 2 * Math.PI);
ctx.stroke();
This is the simplest implementation you can find. That's all the CSS and HTML you will need, which is deceivingly relieving. ANd the JS bit draws a single 40 radius circle... that's where the fun part of canvas lies.

Overview

Together with Canvas API it can be used for animation, game graphics, data visualization, photo manipulation, and real-time video processing

It's supported by most browsers, starting from IE 9 but not all features, so be mindful and check before using. Keep in mind that canvas element and canvas api are like conjoined twins - separate entities but not really

First off...

Canvas vs. SVG / CSS / DOM
Canvas charts, graphs, dynamic diagrams with less memory use but a wall of code
SVG existing shapes to render a sweet tandem with Adobe Illustrator
CSS & DOM large static areas to animate or 3D transforms
... but before we nose dive into that, I wanted to mention anything BUT games. What should you use when creating dynamic content for your site... 1) Canvas is lower level than those others so you can have more control over the drawing and use less memory, but at the cost of having to write more code. 2) Use SVG when you have existing shapes that you want to render to the screen, like a map that came out of Adobe Illustrator. 3) Use CSS or DOM animation when you have large static areas that you wish to animate, or if you want to use 3D transforms.

2D Games

WebGL vs. Canvas API

2D and 3D are interlinked, in a way, but have colossal differences what it comes to development. And yes, there are numerous 2D libraries and engines to choose from aside from vanilla, but bear down with me to basics for now. . . The Canvas API largely focuses on 2D graphics. The WebGL API, which also uses the canvas element, draws hardware-accelerated 2D and 3D graphics.

WebGL

Basically, WebGL is a Javascript API ( based on OpenGL ) that allows us to render interactive 2D and 3D graphics

Version 1.0 | Version 2.0

WebGL 1.0 was released in 2011 - Exposed 3D rendering to the Web Platform for the first time Technical issues such as known hardware problems or lack of required GPU features may prevent WebGL from running in some cases.

Version 1.0

. . . is supported through out browsers on both desktop and mobile platforms.

Chrome, Firefox, Internet Explorer, Opera, and Safari.

Version 2.0

But everything changed when an update was released...

. . . Safari declared it won't support any new version of WebGL and is rumored to be developing a native engine, which will release on the same year as Half-Life 3

WebGL 2.0 requires hardware with OpenGL ES 3.0 support or comparable desktop OpenGL feature support. Not all systems capable of running WebGL 1.0 will be able to run WebGL 2.0.

Canvas API vs. WebGL

General criteria for evaluation

  • Integrates with an already built system
  • Easier to use
  • Faster
  • Has project-required capabilities
  • Production cost
  • Platfrom-independant
Is there any reason, except performance, for using WebGL instead of 2D-Canvas for 2D games/apps? In other words what 2D functionalities are offered by WebGL which are not possible to achieve easily with 2D-Canvas? Looking at this question from another side - how does a developer choose one technology over another? So I'll discuss the differences between canvas and webGL APIs regarding these qualities.

Personal view

Integration Both are JS APIs, hence similar integration
Ease With library - Canvas API From scratch - WebGL
Speed WebGL
Capabilities WebGL
Lower cost Canvas API
Platform Similar
The biggest advantage is the programmability of the pipeline, and the performance. For example, say you are drawing 2 boxes one above other and one is hidden, some GL implementations have scope for discarding the hidden box. 2.1) They are both supported by a number of libraries that could speed up your coding. Different libraries give you different ways to organize your code, so library choice dictates how your drawing APIs are structured. If you use a library, the ease of writing code depends on the library itself. 2.2) If you write code from zero, the canvas API is much easier to learn and understand. It requires minimal math knowledge, and development is fast and straightforward. 3, 4) WebGL is faster and it has more capabilities. It's a native 3D API that gives you full access to the rendering pipeline, code and effects are executed faster and are more 'tweakable'. With webGL there really is no limit. 5) Working with the WebGL API requires strong math skills and a full understanding of the rendering pipeline. Hence the production is slower due to the size and complexity of such a code base, and therefore it costs more. 6) Both canvas and webGL are html5 goodies. Usually the devices that support one will support and the other.

Be it Canvas API or WebGl, being buried under a wall of self-made buggy bicycles and crutches is no ones dream com true...

3D Games: + Libraries

The tale of two frameworks

    Babylon.js

    Summer 2013

  • Babylon Native
  • Node material editor

    Three.js

    Spring 2009

  • Modular rendering interface
  • Not limited to WebGL
Babylon.js, being the relative newcomer, broke onto the scene in the summer of 2013. Brought to you by the minds at Microsoft, it was introduced alongside IE 11’s first official support for the WebGL API. Its a powerful rendering engine with augmented, mixed, and virtual reality tools, node material editor not unlike Blender and with Babylon native you can now supply any device with your newest creation. Three.js got its start back in April of 2009 and was originally written in ActionScript before being translated to JavaScript. Having been created before the introduction of WebGL, Three.js has the unique convenience of a modular rendering interface allowing it to be used with SVG and HTML5’s canvas element in addition to WebGL. It was created with one goal in mind: to take advantage of web based renderers for creating GPU enhanced 3D graphics and animations. As such, this framework employs a very broad approach to web graphics without focusing on any single animation niche. This flexible design makes Three.js a great tool for general purpose web animations like logos or modeling applications

VR / AR

And finally theres the newest niche added in early 2014 to our browsers - virtual and augmented reality. Made possible by WebVR, an experimental JavaScript api that enables applications to interact with virtual reality devices, such as the HTC Vive, Oculus Rift or Google Cardboard in a web browser

AR / VR Library

A-Frame is one of several but a personal favorite due to ease of use and a comprehensive documentation

Although some of the previous libraries have had compatibility with VR as an added feature, A-Frame was built with VR and AR in mind. This means the focus of the entire framework is shifted. Of course, that means you can create 3D games with this it, but if you’re using WebXR compatible browsers or even the experimental features in the latest versions of Chrome that provide support for WebXR AR you’ll be able to make the most out of this framework. Since A-Frame was designed to be used on the browser, it has a very mark-up based syntax, so instead of relying heavily on JavaScript, it also uses some custom markup elements. And let's agree, looks pretty darn cool.
